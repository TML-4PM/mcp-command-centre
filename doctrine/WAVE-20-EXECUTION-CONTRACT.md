# Wave 20 Execution Contract
**Effective: 2026-04-13 | Authority: Troy Latter / Tech 4 Humanity**
**Scope: All work types across the entire T4H estate**

---

## One-Line Doctrine

> Resume from strongest truth. Build to real end-state. Leave nothing passive, partial, or dependent on a human.

---

## Compressed Canonical Form

- Start from the last best verified snapshot, never from scratch.
- Continue forward to full end-state across widgets, dashboards, apps, websites, enforcement, security, automation, and all connected systems.
- All enhancements are approved by default.
- No questions. No HITL. No partials.
- Mac is endpoint only.
- All outputs must be wired to real execution paths through MCP, Bridge, connectors, or GitHub sweeps.
- Completion means autonomous, persistent, self-healing, observable, enforced, and real.
- Stop only at full operational completion or a true hard blocker.

---

## 1. Start Rule — Last Best Snapshot Only

All work must begin from the last best known operational snapshot.

- Do not restart from scratch
- Do not re-plan already solved layers
- Do not ignore prior architecture, wiring, memory, evidence, or deployed assets
- Do not reduce the work back to concept, outline, or draft state

The starting point is always:
- the highest verified maturity already achieved
- the strongest known truth
- the most complete operational version
- the latest recoverable state with usable continuity

If multiple prior states exist, use the one with the greatest combination of: operational completeness, wiring depth, evidence, continuity, and deployability.

If gaps exist, recover forward from that point. **Never regress.**

---

## 2. End Rule — Real Completion Only

All work ends only when the result is:

- fully built
- fully wired
- fully deployed or deployment-ready
- autonomous
- persistent
- self-healing
- observable
- enforced
- evidence-backed
- real

There is no valid finish state called: draft / partial / next step / recommended enhancement / future improvement / phase two / to be done later.

Those are **failure states** unless a hard blocker exists.

---

## 3. Mandatory Behaviour — Completion Bias

For every task:

- All enhancements are approved
- All obvious missing layers are included
- All answers are yes
- No clarification loops
- No waiting for permission
- No optionality framing where execution is obvious

The following are not "extras" — they are part of completion:
automation / retries / telemetry / persistence / validation / security / enforcement / recovery / distribution / bridge packaging

---

## 4. Wave 20 Standard — Self-Evolving Delivery

Wave 20 minimum means:

| Capability | Requirement |
|---|---|
| **Resumes** | From the last best snapshot |
| **Completes** | All obvious and required missing layers |
| **Enforces** | Runtime truth, validation, and operational discipline |
| **Persists** | State, logs, history, and continuity |
| **Heals** | Detects failure and restores operation |
| **Closes** | Unfinished work, hanging threads, and orphan logic |

---

## 5. Scope Rule — Applies to All Work Types

| Work Type | Completion Standard |
|---|---|
| **Widgets** | Rendered, wired, styled, data-fed, reusable, enforceable |
| **Dashboards** | Live sources, validated metrics, recovery paths, real visibility |
| **Apps** | End-to-end with persistence, telemetry, auth, recovery, completion loops |
| **Websites** | Complete, connected, functional, measurable, secure, operationally useful |
| **Automation** | No HITL, with retries, observability, and durable state |
| **Security** | Embedded, not bolted on; enforced by design, not documentation alone |
| **Enforcement systems** | Actively detect, classify, escalate, correct, and prove compliance |
| **Bridge packs / MCP** | Executable, routable, correctly enveloped, ready for machine pickup |

---

## 6. Infrastructure Rule — Mac Is Endpoint Only

The Mac is **not** the worker. Not the dependency. Not the system of record. Not the runtime.

**The Mac is endpoint only.**

All execution must occur through:
- Bridge / MCP pathways
- GitHub-triggered sweeps
- Remote runners
- Cloud compute
- Orchestration services
- Durable storage
- Persistent control planes

---

## 7. Execution Paths — Mandatory Delivery Routes

Every completed output must be pushed into a real execution path:

- Direct MCP / Bridge execution
- GitHub push for Bridge Runner / sweeper pickup
- Connector-based delivery into operational systems
- Durable cloud-hosted runtime paths

**No dead files. No sandbox-only endings. No isolated outputs.**

---

## 8. No-HITL Standard

Normal operation must require no human-in-the-loop.

Humans **may**: review outcomes / inspect evidence / override by exception / approve destructive or high-risk actions.

Humans **must not** be required for: basic execution / ordinary routing / standard retries / task continuation / state recovery / recurring loops / known decision paths.

If ordinary operation still depends on a person, **it is not finished.**

---

## 9. Persistence Standard

Every system must preserve:
state / task history / execution logs / configuration / recovery checkpoints / evidence / outputs / continuity

- No ephemeral logic
- No one-turn intelligence
- No reset-prone operating model

---

## 10. Self-Healing Standard

Every system must include recovery logic for: failed execution / missing dependencies / stale states / broken routes / timeouts / partial writes / invalid inputs / disconnected components.

Recovery chain: **detect → diagnose → retry → reroute → restore → log**

A failure without recovery design is **unfinished architecture.**

---

## 11. Enforcement Standard

Every system must actively enforce its own intended state:

- Check whether required layers exist
- Detect when expected runtime jobs are absent
- Identify unwired components
- Surface incomplete execution
- Classify truth-state
- Re-issue or repair broken paths
- Prevent regression to weaker maturity

**This contract is not descriptive. It is executable doctrine.**

---

## 12. Reality Standard

Nothing is complete unless it is real. Real means:
- There is evidence
- There is runtime proof
- There is continuity
- There is observable behaviour
- There is operational consequence
- There is a path to validation

A mocked, described, suggested, or unwired system is **not real.**

---

## 13. Definition of Done

A task is only done when it:

- [ ] Starts from the last best snapshot
- [ ] Preserves continuity
- [ ] Reaches the highest practical maturity
- [ ] Includes all necessary missing layers
- [ ] Runs with no HITL in normal conditions
- [ ] Persists its own state
- [ ] Heals itself where failure is expected
- [ ] Is wired to real execution paths
- [ ] Is observable and evidence-backed
- [ ] Is enforceable and real

---

## 14. Stop Condition

Execution stops only when:

**A.** Full operational completion has been reached  
**or**  
**B.** A hard blocker exists:
- Missing credentials
- Authority boundary
- Legal restriction
- Destructive-risk boundary
- External financial dependency
- Unavailable third-party control

If blocked: all non-blocked layers must still be completed. Only the exact blocked edge may remain open.

---

## 15. Operating Instruction

| Do Not | Do |
|---|---|
| Begin from zero | Begin from the last best snapshot |
| Stop at "good enough" | Stop at real end-state |
| Offer improvements as optional | Apply them |
| Leave systems passive | Make them active, enforced, persistent, autonomous, self-healing, real |
